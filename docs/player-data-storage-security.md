# 플레이어 데이터 저장 및 보안

> 상태: 설계 기준
> 대상 데이터: 도감, 업적, 경험치, 사냥터 보관함, 캐릭터 연계, 메타 성장, 도전 단계, 사용자 설정
> 현재 배포: GitHub Pages 정적 호스팅

## 1. 결론

현재 프로젝트는 서버가 없는 GitHub Pages 게임이므로 플레이어 진행 데이터는 **`localStorage`** 에 저장합니다.

쿠키는 진행 데이터 저장소로 사용하지 않습니다.

현재 구조에서 RSA, AES, HMAC 또는 해시를 추가해도 사용자가 자신의 브라우저 데이터를 조작하는 것을 확실히 막을 수 없습니다. 모든 검증 코드와 키가 브라우저에 배포되기 때문입니다.

진짜 부정 방지가 필요해지는 시점에는 다음 구조로 전환합니다.

```text
브라우저
  -> HttpOnly 세션 쿠키로 인증
  -> 서버 API에 진행 이벤트 전송
  -> 서버가 결과 검증
  -> 서버 데이터베이스를 최종 원본으로 저장
```

## 2. 위협 모델

### 현재 보호 대상

- 저장 데이터가 깨졌을 때 게임이 실행되지 않는 문제
- 이전 버전 데이터와 새 버전 데이터의 불일치
- 잘못된 숫자로 인한 무한 루프, `NaN`, 과도한 스탯
- 다른 GitHub Pages 프로젝트와 저장 키가 충돌하는 문제
- XSS로 인한 데이터 노출 가능성

### 현재 보호할 수 없는 대상

- 사용자가 개발자 도구에서 자신의 진행 데이터를 수정하는 행위
- 사용자가 JavaScript 실행을 중단하거나 검증 함수를 우회하는 행위
- 사용자가 수정한 게임 코드를 로컬에서 실행하는 행위
- 브라우저 프로필 파일에 직접 접근할 수 있는 프로그램

현재 게임은 계정, 거래, 경쟁 순위, 유료 재화가 없는 싱글플레이이므로 로컬 조작은 해당 사용자 자신의 경험에만 영향을 줍니다. 이 단계에서는 강한 부정 방지보다 데이터 안정성과 복구 가능성을 우선합니다.

## 3. 저장 방식 비교

| 기준 | localStorage | 쿠키 | IndexedDB | 서버 저장 |
| --- | --- | --- | --- | --- |
| 정적 GitHub Pages에서 사용 | 적합 | 제한적 | 적합 | 별도 서버 필요 |
| 브라우저 종료 후 유지 | 가능 | 만료 설정 필요 | 가능 | 가능 |
| 구조화된 소형 JSON | 적합 | 부적합 | 가능 | 적합 |
| 대용량 데이터 | 부적합 | 매우 부적합 | 적합 | 적합 |
| 요청마다 서버 전송 | 안 함 | 자동 전송 | 안 함 | API 요청 시 전송 |
| JavaScript 접근 | 가능 | 일반 쿠키는 가능 | 가능 | 응답 데이터만 가능 |
| 사용자 조작 방지 | 불가능 | 일반 쿠키는 불가능 | 불가능 | 서버 검증 시 가능 |
| HttpOnly 지원 | 해당 없음 | 서버 설정 시 가능 | 해당 없음 | 세션 쿠키에 사용 |

### localStorage 선택 이유

- 현재 진행 데이터는 작은 JSON입니다.
- 브라우저를 닫았다 다시 열어도 유지되어야 합니다.
- 동기 API지만 저장 횟수가 토너먼트 결과 시점 정도로 적어 성능 문제가 작습니다.
- 쿠키처럼 모든 HTTP 요청에 자동으로 포함되지 않습니다.
- 현재 필요한 데이터 크기는 일반적인 localStorage 한도보다 충분히 작습니다.

### 쿠키를 선택하지 않는 이유

- 쿠키는 서버와 세션 상태를 주고받기 위한 용도에 가깝습니다.
- 일반적인 쿠키는 사용자가 보고 수정할 수 있습니다.
- `HttpOnly` 쿠키는 서버의 `Set-Cookie` 응답이 있어야 하며 JavaScript에서 읽을 수 없습니다.
- 현재 게임에는 프로필을 읽고 갱신할 서버가 없습니다.
- 쿠키는 일반적으로 개별 크기가 작고 요청마다 전송되어 진행 데이터 저장에 비효율적입니다.

### IndexedDB 전환 기준

다음 데이터가 추가되면 IndexedDB를 검토합니다.

- 전투 리플레이
- 대량의 매치 히스토리
- 대량의 사냥터 런 히스토리 또는 상자 개봉 로그
- 이미지 또는 사용자 생성 콘텐츠
- 수천 건 이상의 이벤트 로그
- 비동기 검색과 부분 갱신이 필요한 데이터

도감과 업적 카운터만 저장하는 동안에는 localStorage가 더 단순합니다.

## 4. 저장 키

```js
export const PLAYER_PROFILE_STORAGE_KEY = "bfs:player-profile:v1";
```

GitHub Pages의 여러 저장소 사이트는 `https://byh020907.github.io`라는 같은 origin을 공유할 수 있으므로 프로젝트 접두사 `bfs`를 반드시 포함합니다.

키 이름 변경은 데이터 마이그레이션 없이 진행하지 않습니다.

## 5. 저장 모듈 책임

```text
src/player-profile.js
  loadPlayerProfile()
  savePlayerProfile(profile)
  migratePlayerProfile(raw)
  sanitizePlayerProfile(raw)
  createDefaultPlayerProfile()
  exportPlayerProfile()
  importPlayerProfile(text)
```

- 다른 모듈에서 `localStorage`를 직접 호출하지 않습니다.
- 로드한 데이터는 사용 전에 항상 마이그레이션과 보정을 거칩니다.
- 저장은 매 프레임이 아니라 토너먼트 결과, 설정 변경 등 명확한 시점에 수행합니다.
- `QuotaExceededError`, 접근 거부, JSON 파싱 오류를 처리합니다.
- 저장 실패 시 메모리 상태로 게임은 계속 진행하고 사용자에게 저장 실패를 알립니다.

## 6. 데이터 보정

클라이언트 저장 데이터는 신뢰하지 않습니다.

```js
const PROFILE_LIMITS = Object.freeze({
    MAX_COUNTER: 1_000_000_000,
    MAX_TIMESTAMP: 8_640_000_000_000_000
});
```

로드 시 적용할 규칙:

- 알 수 없는 객체와 필드는 필요한 경우 무시합니다.
- 숫자는 유한한지 확인하고 음수 카운터를 0으로 보정합니다.
- 배열은 중복을 제거하고 등록된 ID만 유지합니다.
- 캐릭터 연계 ID는 등록된 효과만 유지하고 중복을 제거합니다.
- 도전 단계는 최고 해금 범위 안으로 보정합니다.
- 성장 보너스는 각 상한으로 제한합니다.
- 타임스탬프가 잘못되면 `null`로 복구합니다.
- 프로필 객체를 그대로 도메인 객체로 사용하지 않고 필요한 형태로 새로 만듭니다.

이 검증은 부정 방지가 아니라 게임 안정성을 위한 것입니다.

등록된 연계 ID를 사용자가 개발자 도구로 직접 추가한 경우, 현재 클라이언트 전용 구조에서는 정상 해금과 구분할 수 없습니다. 따라서 문서와 테스트에서 `미해금 등록 ID 차단`을 보안 요구사항으로 두지 않습니다.

## 7. 암호화와 서명의 한계

### 7.1 암호화

암호화는 내용을 숨기는 **기밀성**을 위한 기능입니다. 값이 정당하게 생성됐는지를 보장하는 기능이 아닙니다.

- RSA 공개키로 데이터를 암호화하면 브라우저는 서버의 개인키 없이 다시 읽을 수 없습니다.
- 브라우저가 읽기 위해 개인키를 함께 배포하면 사용자가 그 키를 추출할 수 있습니다.
- AES 키를 브라우저 코드나 localStorage에 넣어도 같은 문제가 생깁니다.
- 난독화와 Base64는 암호화가 아닙니다.

따라서 로컬 진행 데이터 전체를 RSA로 암호화하지 않습니다.

### 7.2 전자서명

전자서명은 서버가 개인키로 데이터에 서명하고 브라우저가 공개키로 검증할 때 의미가 있습니다.

```text
서버 개인키: 서명
브라우저 공개키: 검증
```

브라우저가 오프라인에서 진행 데이터를 스스로 갱신하려면 새 서명을 만들 수 있어야 합니다. 서명용 개인키를 브라우저에 넣는 순간 사용자가 임의 데이터에도 서명할 수 있으므로 보호가 사라집니다.

### 7.3 HMAC과 해시

- SHA-256 해시는 데이터와 함께 저장하면 사용자가 데이터를 바꾼 뒤 해시도 다시 계산할 수 있습니다.
- HMAC은 비밀키가 필요하지만 그 키를 브라우저에 배포하면 사용자가 추출할 수 있습니다.
- 해시는 우발적인 손상 감지에는 사용할 수 있지만 부정 방지 기능이라고 표시하지 않습니다.

## 8. 현재 단계의 변조 대응

서버 없이 적용할 수 있는 대응:

1. 모든 저장값 스키마 검증
2. 상한과 ID 화이트리스트 적용
3. 버전 기반 마이그레이션
4. 저장 실패와 손상 데이터 복구
5. 프로필 내보내기 및 가져오기
6. 사용자에게 데이터 초기화 기능 제공
7. CSP와 안전한 DOM 바인딩으로 XSS 위험 축소
8. 선택적으로 체크섬을 저장해 우발적인 손상 감지

체크섬 불일치 시 자동으로 전체 진행을 삭제하지 않습니다. 백업 복원 또는 초기화 여부를 사용자에게 안내합니다.

## 9. 서버 전환 기준

다음 기능 중 하나라도 추가되면 서버 권위 저장을 도입해야 합니다.

- 온라인 리더보드
- 다른 사용자와 비교되는 랭킹
- PvP 또는 협동 플레이
- 거래와 선물
- 유료 재화 또는 결제 보상
- 계정 간 클라우드 동기화
- 공식 대회 기록
- 조작된 진행 데이터가 다른 사용자에게 영향을 주는 기능

### 권장 서버 구조

```text
브라우저
  -> HTTPS
  -> Secure + HttpOnly + SameSite 세션 쿠키
  -> 진행 이벤트 API
  -> 서버 검증
  -> 데이터베이스
```

- 쿠키에는 진행 데이터가 아니라 추측하기 어려운 세션 식별자만 저장합니다.
- 진행 데이터의 최종 원본은 서버 데이터베이스입니다.
- 서버는 클라이언트가 보낸 최종 숫자를 그대로 저장하지 않고 가능한 경우 원시 이벤트를 검증합니다.
- RSA 서명이 필요하다면 개인키는 서버에만 둡니다.
- 브라우저 localStorage는 오프라인 캐시나 마지막 UI 상태 용도로만 사용합니다.

## 10. 프로필 내보내기

현재는 계정 동기화가 없으므로 사용자가 브라우저를 바꿀 때 진행을 옮길 수 있는 수단이 필요합니다.

```js
{
    product: "ball-fight-simulator",
    exportedAt: 1719000000000,
    profile: {}
}
```

- 가져오기 전에 동일한 마이그레이션과 보정 과정을 거칩니다.
- 가져온 데이터도 신뢰하지 않습니다.
- 내보내기 파일에 인증 토큰이나 개인 정보를 넣지 않습니다.
- 암호화되지 않은 게임 진행 정보임을 사용자에게 알립니다.

## 11. 필수 회귀 조건

- localStorage가 비어 있으면 기본 프로필을 반환해야 합니다.
- JSON이 깨져 있어도 게임이 시작되어야 합니다.
- 잘못된 숫자와 알 수 없는 ID가 도메인 상태에 들어가지 않아야 합니다.
- 저장 실패가 전투 루프를 중단하지 않아야 합니다.
- 다른 모듈이 localStorage를 직접 호출하지 않아야 합니다.
- 프로필 내보내기 후 다시 가져오면 보정된 동일 상태를 복원해야 합니다.
- 버전이 오래된 프로필은 순서대로 마이그레이션되어야 합니다.
- 프로필 키에 `bfs` 프로젝트 접두사가 포함되어야 합니다.
- 체크섬이 존재하더라도 보안 서명으로 취급하지 않아야 합니다.

## 12. 참고 근거

- [MDN Web Storage API](https://developer.mozilla.org/en-US/docs/Web/API/Web_Storage_API): localStorage는 origin 기준으로 공유되며 브라우저를 닫았다 열어도 유지됩니다.
- [MDN HTTP Cookies](https://developer.mozilla.org/en-US/docs/Web/HTTP/Guides/Cookies): `HttpOnly`와 `Secure`는 서버 세션 쿠키 보호에 사용합니다.
- [RFC 6265](https://www.rfc-editor.org/info/rfc6265/): 쿠키는 크기 제한이 있고 모든 요청의 Cookie 헤더에 포함되므로 작게 유지해야 합니다.
- [MDN Web Crypto API](https://developer.mozilla.org/en-US/docs/Web/API/Web_Crypto_API): 암호 기능 자체보다 키 관리와 전체 보안 설계가 어렵고 잘못 사용하기 쉽습니다.
- [OWASP Browser Storage](https://owasp.org/www-project-web-security-testing-guide/latest/4-Web_Application_Security_Testing/11-Client-side_Testing/12-Testing_Browser_Storage): 브라우저 저장소는 개발자 도구와 로컬 권한으로 확인 및 변경할 수 있으므로 기밀성이나 권위 저장소로 가정하면 안 됩니다.
